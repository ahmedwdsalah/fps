### intro to training data?



- **1,188,265 one-dimensional numeric arrays** collected from 5 real-world domains
- For every array, extracted **16 structural features**, all computed in O(n) time — meaning only scan the array once, which is cheaper than sorting itself (O(n log n)), so the feature extraction adds negligible overhead

- **timed 3 sorting algorithms** (introsort, heapsort, timsort) on each array to record which one finishes fastest — this is the **labeling step** (the fastest algorithm becomes the "correct answer" the model learns from). The labeled dataset is saved in `data/training_dataset.csv` (1.18M rows × 22 columns). Domain split: F1 74.5%, Stock 8.4%, Crypto 8.4%, Earthquake 8.4%, Weather 0.3%

  - **Timsort** drops to **O(n)** on already-sorted or nearly-sorted data — it detects natural runs and merges them ( like merage sort, and this is what most of real world data is)
  - **Introsort** is fast on random data but degrades on adversarial patterns — it switches to heapsort internally when recursion gets too deep
  - **Heapsort** is the most consistent — always O(n log n) with no best/worst case variance, but higher constant factor due to cache-unfriendly memory access

### Why these 5 domains?

The goal is**structural diversity**, not domain specific. Different real-world domains produce arrays with different statistical shapes:

- **F1 telemetry:** partially-sorted signals (speed increases on straights, decreases at corners) — **timsort** win cuz the "natural runs".
- **Stock/Crypto:** random, a mix where **heapsort** and **introsort** has more to explore.
- **Earthquake:** distributed magnitudes, bimodal depths, heavy clustering — produces **outlier-heavy, skewed** arrays unlike any other domain.
- **Weather:** high autocorrelated signals — nearly sorted, **timsort** win.

### data store and processing?

1. **Raw arrays:** Each array saved as an individual CSV file under `data/real_world_10k/raw/`, organized by domain. Total size ~12 GB.

2. **Structural transforms:** To prevent bias, applied 4 transforms to every raw array:
   - **REV** — reversed the array (forces worst-case for timsort)
   - **SHUF** — random shuffle (pure disorder, favors heapsort/introsort)
   - **QBIN50** — quantized to 50 bins (creates massive duplicates)
   - **PSORT10** — sorted then perturbed 10% of elements (near-sorted)

   This gives 5× the array count (RAW + 4 transforms) and ensures all 3 algorithms get fair representation.

3. **Timing:** For each array, we ran all 3 sorting algorithms and recorded wall-clock time (best-of-N repeats, garbage collection disabled, `perf_counter` precision). N=5 for arrays ≤10K elements, N=3 for larger arrays.

4. **Feature extraction:** 16 features computed per array using `scripts/feature_extraction.py` (single source of truth, imported by all training scripts). All features are O(n) except `inversion_ratio` which is O(n log n) but subsampled to 2000 elements for large arrays.


### Raw class distribution (before balancing)

| Algorithm | Wins | % |
|-----------|------|---|
| Timsort | 1,010,413 | 85.0% |
| Heapsort | 125,218 | 10.5% |
| Introsort | 52,634 | 4.4% |

This heavy timsort skew is **genuine** — real-world data is mostly partially sorted. We address this imbalance during model training (margin filter + undersampling (method used: random sampling) + class weights)
