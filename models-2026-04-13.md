### why XGBoost?

- gradient-boosted decision trees — fast, handles tabular data well, no need for deep learning on 16 features
- built-in support for multi-class classification, sample weights, early stopping
- interpretable — gives feature importance rankings directly

### why classification not regression?

- **v1 tried regression** — 3 separate models predicting absolute time for each algorithm, then pick the fastest
- failed (51% accuracy) because regression learns array size dominates timing, not which algorithm is relatively faster
- **classification learns decision boundaries** — "for this combination of features, algorithm X wins" — directly what needed

### v5 — production model details

- dataset: 1.18M arrays, after balancing → ~196K training rows
- balancing pipeline:
  1. **margin filter** — remove arrays where best/second-best gap <5% (noisy labels) — keep all arrays with n ≥ 2000 regardless
  2. **random undersampling** — cap each class at 3× minority count
  3. **inverse-frequency class weights** — rare classes (introsort) weigh more in loss function
- hyperparameters: 500 trees, max_depth=7, learning_rate=0.05, subsample=0.8
- split: 70% train / 15% val / 15% test (stratified)
- eval metric: mlogloss

### v8 key finding — introsort ≈ heapsort

- stage 1 (timsort vs rest): AUC = 0.982 — easy, clearly separable
- stage 2 (introsort vs heapsort): AUC = 0.603 — barely above coin flip
- 16 O(n) features **cannot distinguish** introsort from heapsort
- the problem is effectively **binary** (timsort vs not-timsort)
- no further offline model gains possible with current features — ceiling reached

### what v3 proves

- v3 added 6 timing-derived features (log times + pairwise differences between algorithms)
- accuracy jumped to 95% — proves the signal exists, just not in structural features alone
- this is the **ceiling** the bandit (step 5) aims toward — learn timing patterns online without needing all 3 timings upfront
